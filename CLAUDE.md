# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A personal insulin/glucose/activity tracker for type 1 diabetes management. It polls live glucose data from LibreLinkUp (Freestyle Libre follower API), accepts Apple Health activity data via Apple Shortcuts webhooks, and does pattern analysis on insulin corrections. Not medical software — single-user. Deployed to Render (see `README.md` "Deploy to Render"); the original design target was Glitch and that path still works.

## Commands

- `npm start` — runs `server.js` (only script defined; no build step)
- No test suite, no linter/formatter configured (no jest/vitest, no eslint/prettier)
- Requires Node >= 18 (uses global `fetch`)

## Running locally

Create a `.env` file in the project root (loaded manually by `lib/config.js`, not via dotenv package):
```
LLU_EMAIL=...
LLU_PASSWORD=...
LLU_REGION=EU          # optional, default EU
APP_PASSWORD=...
PORT=3000               # optional, default 3000
# UPSTASH_REDIS_REST_URL / UPSTASH_REDIS_REST_TOKEN  # optional, see Data persistence below
```
The server calls `process.exit(1)` at startup if `LLU_EMAIL`/`LLU_PASSWORD`/`APP_PASSWORD` are missing or still set to placeholder values. Note `.env` is gitignored — a separate deploy host (Render, Glitch) needs these set in its own env var UI; they don't travel with the repo.

## Architecture

Backend is `server.js` (a thin Express entry point) over five `lib/` modules; frontend is three plain files in `public/` (no framework, no bundler), served statically and rendered as a single-page app with 5 tabs (Track / Activity / Insights / Today / Settings):

- **`lib/config.js`** — manual `.env` loading, constants (`DEFAULT_SETTINGS`, `MIN_PLAUSIBLE_FACTOR`, `MIN_EFFECTIVE_CORRECTION_UNITS`, etc.), startup validation (`process.exit(1)` on missing creds).
- **`lib/store.js`** — `loadData()`/`saveData()` single-blob persistence (Upstash Redis or local `data.json`), `backfill()`.
- **`lib/libre.js`** — LibreLinkUp client: `login()`, `fetchGlucose()`, `computeTrend()`, `parseLibreTimestamp()`, `backfillFromGraph()`, the glucose cache, `POLL_MS`.
- **`lib/analysis.js`** — the domain logic: `resolveCorrections()`, `analysePatterns()`, `suggestMealDose()`, `getDailySummary()`, `checkInsulinHealth()`, the forward glucose simulation (`simulateForward()` and friends), forecast scoring (`scoreForecasts()`/`forecastAccuracy()`), plus `resolveEntryTime()`/`glucoseAt()` helpers.
- **`lib/routes.js`** — all Express routes + `requireAuth`.
- **`public/index.html`** (markup) / **`public/styles.css`** / **`public/app.js`** (all frontend logic, loaded as a plain script — functions are global because the markup uses inline `onclick=`).

`server.js` sets `trust proxy` (Render/Glitch terminate TLS at a proxy, so `req.ip` and `req.secure` need it) and parses JSON up to **5mb** — raised from 1mb because Health Auto Export payloads embed per-minute heart-rate arrays we discard but Express must still parse; a multi-workout export cleared 1mb and 413'd, silently breaking the sync.

One deliberate wiring quirk: `analysis.js` needs the glucose cache from `libre.js`, and `libre.js` needs to trigger work after each fetch — that direction is injected (`setOnGlucoseFetched(...)` in `server.js`) rather than required, to avoid a circular dependency. The hook now runs `resolveCorrections()` **then** `scoreForecasts()`, sequentially and never in parallel — both do a read-modify-write of the same blob and must not interleave. If you add another "do X when a reading arrives" behavior, extend that hook rather than adding a require cycle.

### Data persistence

No real database — a single JSON blob read/written wholesale via `loadData()`/`saveData()` in `lib/store.js` (both `async`, all call sites use `await`). Shape:
```js
{ boluses: [], basalDoses: [], corrections: [], activities: [], glucoseHistory: [], forecasts: [], settings: { targetLow, targetHigh, carbRatio } }
```
`backfill()` merges in `DEFAULT_SETTINGS` (and empty `corrections`/`forecasts` arrays) for blobs saved before a field existed, so old data doesn't need a migration.
Retention is pruned inline: `glucoseHistory` kept 14 days (extended from 7 so `checkInsulinHealth()`'s week-over-week comparison has a prior week of glucose to read — at 7 days the "prior week" window was always empty and that comparison silently never fired), `activities` kept 30 days, `forecasts` kept 14 days (sampled every ~30min, not on every 60s client poll — see Forecast scoring), `corrections` kept indefinitely (needed for long-term pattern analysis). `analysePatterns()` deliberately re-scopes `glucoseHistory` down to just the trailing 7 days of that for its own insights, so its "over the last 7 days" language stays accurate and day-of-week-style patterns (which would need many more weeks to be meaningful) aren't attempted.

Storage backend depends on environment:
- If `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN` are set, the blob is stored as a single key in Upstash Redis via plain `fetch()` against its REST API (no client library, same style as the LibreLinkUp calls). This is what Render (no persistent disk on the free tier) uses in production.
- Otherwise, falls back to a local `data.json` file: `.data/data.json` if a `.data` dir exists (Glitch convention), else project root. This is what local dev and Glitch use.

When adding new data fields or call sites, remember `loadData()`/`saveData()` are async now — a missed `await` will silently race against the Upstash HTTP round-trip.

### Auth

Single shared `APP_PASSWORD` (no user accounts). `requireAuth` middleware in `lib/routes.js` accepts either an httpOnly cookie (`auth`, sha256 of the password, set via `POST /api/login`) or a `Bearer <APP_PASSWORD>` header — the latter path exists specifically so Apple Shortcuts can POST to `/api/health` without cookie support.

`POST /api/login` is rate-limited per IP (in-memory `loginAttempts` map): 5 wrong attempts, then a 60s lockout returning **429** with the remaining wait; a correct password clears the counter immediately so a legitimate fat-finger doesn't compound. The map resets on restart, which is fine for a single-user app. The auth cookie sets `secure: req.secure` rather than a hardcoded `true` — Secure on Render/HTTPS, still working over plain HTTP in local dev. The login screen shows the server's actual error text so a locked-out user sees "wait 60s" instead of "wrong password".

### LibreLinkUp integration

`lib/libre.js` maintains its own auth session against the LibreLinkUp API (`REGIONS` map of per-region base URLs, spoofed iOS `User-Agent`/`product` headers to match the official app). Key pieces:
- `login()` — authenticates, handles the "Terms of Use" re-acceptance redirect flow, caches `authToken`/`tokenExpiry` (50 min).
- `fetchGlucose()` — cached per `POLL_MS` (5 min, intentionally matched to the official app's poll rate to avoid account restrictions — do not lower this); on a genuinely new reading (>3 min since last stored point) appends to `glucoseHistory` and triggers the fetch hook (see Architecture above). Uses `FactoryTimestamp` (UTC) rather than `Timestamp` (the account's unmarked local time).
- **`parseLibreTimestamp()` — read this before touching any reading-age code.** `FactoryTimestamp` is UTC but arrives with **no timezone designator** (`"7/23/2026 7:13:45 AM"`), so `new Date(str)` parses it in whatever local zone the runtime happens to be — server *or browser*. That's harmless for a *difference* of two such times (the delta chip, `computeTrend()`: the offset cancels), but the reading's **age** is not a difference, so every fresh reading looked ~1h stale on a UK phone and the sensor-gap warning fired constantly. `parseLibreTimestamp()` parses the `M/D/YYYY h:mm:ss AM/PM` components explicitly as UTC. The parsed epoch ships on the payload as **`readingMs`**, and the client and `dosingContext()` age readings off *that* — never by re-parsing the raw string. (This retires the old "only correct on Render/UTC" caveat: age is now right on any timezone.)
- `backfillFromGraph()` — seeds history from `/llu/connections/{patientId}/graph` (~12h of ~15-min-spaced points; the `/llu/connections` list endpoint does **not** carry `graphData`). Runs only when the trailing 6h holds fewer than 3 stored readings — i.e. a cold start, redeploy, or long gap — so steady-state polling adds **no** extra API calls. Points within 4min of an existing reading are skipped, making it idempotent; backfilled points carry `trend: null` (no consumer needs it, and it's derived from adjacent live readings anyway). Because it can add history without a *new* live reading, the save is gated on a `historyChanged` flag rather than the new-reading branch alone.
- `computeTrend()` derives the rising/falling arrow from the delta between the last two stored `glucoseHistory` points (mmol/L per minute, bucketed), overriding LibreLinkUp's own `TrendArrow` — this is more responsive than the API's arrow without polling more often. Falls back to the API's arrow when there's no usable prior point (cold start or a >20min gap).
- `fetchGlucose()` also exposes the raw (non-bucketed) `delta`/`deltaMinutes` between those same two points on the `/api/glucose` response — shown as a small corner note on the frontend's glucose card (e.g. "+0.3 (5m)"). `null` if there's no prior point or the gap exceeds 20min, same guard as `computeTrend()`.
- Server polls glucose automatically via `setInterval` at startup — this happens independent of any HTTP request.

### Correction-resolution flow (the core domain logic)

When a user logs an insulin "correction" dose (`POST /api/entries/correction`), the entry is stored unresolved (`actualGlucose: null`). On every subsequent glucose fetch, `resolveCorrections()` checks each pending correction:
- Waits at least 2.5h, gives up (marks unresolved) after 4h, setting a `giveUp` flag distinct from `actualGlucose`/`resolved` (which stay `null`/`false`) so a timed-out correction stops being re-evaluated on every future poll - without it, `actualGlucose` never becomes non-null, so the skip-check never excludes it and it gets needlessly re-processed and re-saved every ~5min forever.
- Finds the glucose reading closest to the 3h mark and records it as `actualGlucose`, deriving `dropPerUnit` and `accuracy` vs. the user's `predictedGlucose`.
- **Carbs are split by whether they were knowable at dose time**, because the two cases deserve different treatment:
  - **Later carbs** (an unplanned snack logged *after* the correction, inside the window) — timing and absorption weren't known when dosing, so the drop can't be attributed. Flagged `carbInterference` and excluded from the "clean" `resolved` set used everywhere else (correction-factor average, `suggestedDrop`, `analysePatterns()`, `suggestMealDose()`). The linked meal of a combined log is excluded from this sum via `linkedBolusId`.
  - **Concurrent carbs** (`mealCarbs` from a combined log, or carbs logged in the 30min before) — known at dose time and fully inside the window, so they're **netted out algebraically** instead of thrown away: `drop = units*factor − carbs*(factor/carbRatio)` rearranges to `factor = drop / (units − carbs/carbRatio)`, i.e. divide the drop by the *effective correction units* left after carb coverage. Stored as `effectiveUnits` + `carbAdjusted: true`; these still count toward the ≥3 needed to calibrate. Guarded by `MIN_EFFECTIVE_CORRECTION_UNITS` (1.0u) — a dose that was nearly all carb coverage carries no correction signal and would blow the division up exactly like a too-low factor does. With no `carbRatio` to net with, or too little effective insulin, it falls back to `carbInterference: true`.
  - Carbs eaten **>30min before** are deliberately *not* netted — they're already absorbing and therefore already reflected in `startGlucose`.
  - Why this matters: someone who mostly doses at mealtimes could otherwise never accumulate 3 carb-free corrections, so the factor never calibrated and the forecast stayed blind indefinitely. Naively dividing by raw units instead gives a *falsely low* factor (a real case: 2.0 → 1.50), which reads as "my insulin is weak" and pushes every future suggestion **higher** — the dangerous direction.
- **Combined logging** (`POST /api/entries/combined`) records one injection as two linked entries in a single action: the total is split via `carbRatio` into carb coverage + correction, written as a `bolus` (with `mealName`) plus a `correction` carrying `mealCarbs`/`linkedBolusId`. Total units logged always equals the amount injected — the split only changes *attribution*, never the IOB sum. If the leftover correction is <0.5u it's logged as a plain bolus instead (`split: false`). Surfaced as a checkbox on the Meal card that appears **only** when entered units exceed carb coverage (`updateSplitToggle()`), defaulting on and resetting to on after each log. Without this, a meal-time dose taught the meal model a wildly wrong insulin:carb ratio (a real 5u/10g coffee = 0.5 u/g against a usual 0.1–0.2) *and* taught the correction model nothing.
- The rolling average `dropPerUnit` across clean resolved corrections becomes the user's personal correction factor, surfaced via `GET /api/correction-factor` and used to auto-suggest `suggestedDrop` on future correction entries. That endpoint also returns `recentCarbs` (carbs logged in the last 2h) as a heads-up that the correction may be less predictable.
- `GET /api/correction-factor` also proactively suggests a correction dose: `suggestedUnits = (currentGlucose - settings.idealTarget) / factor`, rounded to the nearest 0.5u, `0` if already at/below target. This is distinct from `suggestedDrop` above - one predicts the outcome of a chosen unit amount, the other suggests the unit amount in the first place. Needs both a personal correction factor (≥3 clean resolved corrections) and `settings.idealTarget` set.
- **All dose suggestions are IOB-, trend-, and staleness-aware via `dosingContext()`** (in `lib/analysis.js`): active insulin (server-side copy of the client IOB model — if you tune `IOB_PEAK`/`IOB_DIA`, change *both* `lib/analysis.js` and `public/app.js` together) is subtracted from suggested units, so a dose already working can't be stacked on; the raw reading is replaced by a ~30-min trend projection (full credit when falling, **half** when rising — a lingering high is correctable later, an over-dosed low is not; clamped ±2 mmol/L so a noisy two-point delta can't swing it); and readings older than 30min set `stale: true`, which the UI surfaces as a "check your Libre app first" warning. COB is deliberately *not* credited back against the subtracted IOB — the net bias is always toward suggesting less insulin. When IOB alone covers the elevation, `coveredByIOB: true` replaces a dose number with "the X.Xu still active should cover it". Staleness age comes from `readingMs` (see `parseLibreTimestamp()` above), so it is correct on any timezone.
- `MIN_PLAUSIBLE_FACTOR` (0.5 mmol/L/unit) guards both this and `suggestMealDose()`'s correction add-on below: a factor averaged from only 3-4 "clean" resolved corrections can land implausibly low if those particular corrections happened not to do much (unrelated glucose noise, illness, etc.), and dividing a real glucose elevation by a near-zero factor produces an absurd suggestion (a 4mmol/L elevation over a 0.1 factor "suggests" 40+ units - this actually happened before the guard existed). Below that floor, `factorTooLow: true` is returned instead of a number. `MAX_SUGGESTED_UNITS` (10) is a hard backstop on top of that regardless - this is a heuristic tool, not medical software, and should never recommend an implausible one-shot dose no matter what the arithmetic says.

`analysePatterns()` (backing `GET /api/insights`) builds on this history: correction-factor accuracy, exercise-day vs. rest-day insulin sensitivity, post-workout glucose deltas, time-of-day highs/lows, and time-in-range (against `settings.targetLow`/`targetHigh`, not a hardcoded range). It also covers: dawn-phenomenon detection, hypo/hyper clustering by 4h time-of-day window, meal-size outcome tertiles, glucose CV, insulin-sensitivity drift (earlier vs. later halves of resolved corrections), basal dose-time drift (>90min spread over 14d; >12h spread is skipped as a deliberate schedule change rather than drift), correction stacking (two doses <2h apart — only flagged once it has actually produced lows), evening-exercise→overnight-low comparison (following night's 00:00–06:00 minimum vs. other nights), and hypo-recovery analysis (episodes of readings <3.9 merged at 30min gaps; reports typical time back above 4.5 and flags when ≥50% of treatments rebound past 10 within 3h).

**Per-workout-type response profiles + live alerts** — `workoutTypeProfiles()` pools every session of the same exact `workoutType` string that has glucose data (full 14-day retention, wider than `analysePatterns()`'s 7-day scope on purpose) and measures baseline (30min pre-start) vs. post-workout nadir over the 8h after finishing — delayed drops land hours later, outside the during-workout window. A profile needs ≥2 sessions with data; a "reliable drop" means `dropShare ≥ 0.6` (share of sessions dropping >1.5 mmol/L) and median drop ≥1.5. Individual per-session insights only render for the last 48h — the profiles carry the longer history, which is what keeps the Insights list from becoming a per-workout log. `getActiveAlerts()` (backing `GET /api/alerts`, polled by the Track tab every 60s alongside glucose) turns a qualifying profile into a *live* heads-up while inside that type's historical drop window after a real workout, plus a stacked-corrections nudge when ≥2 corrections land within 2.5h, and a bedtime check (21:00–02:00) when glucose is modest-and-falling or <7 with >1u still active. Alerts are time-sensitive prompts ("this applies to you right now"); insights are retrospective patterns — don't blur the two.

**Hypo risk forecast — a forward simulation, not an arithmetic projection.** `forecastHypoRisk()` (backing `GET /api/hypo-forecast`, rendered as the persistent card under the glucose display, polled with the 60s cycle and refreshed after log/delete actions).

The old model computed `current + trend − IOB effect`. That **double-counted insulin**: the observed trend *is* the insulin and carbs already acting, so extrapolating it and then subtracting their full effect charged the same insulin twice, biasing every projection low and crying wolf. `simulateForward()` replaces it — glucose is stepped from the **current reading** in `SIM_STEP_MIN` (5min) increments, each input contributing a *rate*:

```
dG = −insulinActing×factor + carbsAbsorbing×(factor/carbRatio) − exerciseDrag + residual
```

The trend is now an **output** of the model, not an input, so nothing is counted twice. Key pieces:

- **Residual (self-correction).** `observedRatePerMin()` fits a least-squares slope over the last 30min of readings (far steadier than the two-point delta `computeTrend()` uses) and the gap between that and the model's own predicted current rate is carried forward, decaying to zero over `RESIDUAL_DECAY_MIN` (60). That absorbs everything unmodelled — a mis-estimated meal, illness, stress — in the near term without degenerating back into pure trend extrapolation.
- **Trough, not endpoint.** The reported `projectedLow` is the **minimum along the simulated path** (with `troughAtMin`), plus `projectedEnd` for the 2h value and the full `path` for the chart. A dip at 45min that recovers by 2h was previously invisible, which is exactly what a hypo warning exists to catch. Note this makes tiering *more* sensitive than the old endpoint-based version.
- **Contextual sensitivity.** `contextualFactor()` prefers the most specific bucket with n≥3 — post-exercise (within 6h of a workout end) first, since it's the largest effect and when hypos actually happen, then the `SENS_BUCKETS` time-of-day bucket, then the global average. The app was already *measuring* all this in `sensitivityMap()` and the forecast was throwing it away for one flat number.
- **Exercise as a rate.** `exerciseDrags()` spreads each open post-workout drop over the remaining time to that type's usual nadir instead of subtracting it as a lump, and **includes in-progress workouts** (negative `hoursSince`, clamped to 0) — during a session the old model was blind.
- **Counter-regulation.** A purely linear model happily projects glucose through zero (a real test produced −1.5). Below `COUNTER_REG_START` (4.5) a falling `dG` is damped toward `GLUCOSE_FLOOR` (2.5), modelling the glucagon/adrenaline response. This does *not* soften the warning — a trough pinned near the floor still reads as deep hypo — it just stops the number being nonsense.
- **Uncertainty band.** `range` widens with distance to the trough and narrows as the factor firms up (wider still when uncalibrated). A single decimal implies precision the model doesn't have.

Tiers: minimal/low/moderate/high off the trough, bumped one tier by ≥2 past hypo episodes in the same 4h time-of-day window, or by uncalibrated-but-insulin-heavy-and-unfed states. Uncalibrated (<3 clean corrections) means no insulin/carb conversion at all — the simulation runs on the decaying residual alone, and `correctionCount` drives the Track tab's calibration banner (`renderCalibration()`) which says so plainly, since the card still lists an IOB figure and the blindness is otherwise invisible. The card always renders (a quiet "minimal" builds the trust that makes "high" actionable); factors expand on tap.

**Forecast scoring (the model's report card)** — `recordForecast()`/`scoreForecasts()`/`forecastAccuracy()`, backing `GET /api/forecast-accuracy` and the 🎯 card on Insights. The forecast used to be fire-and-forget, which made every "improvement" unfalsifiable — **this is the measuring stick, so add it to any model change before trusting the change.** Forecasts are sampled into `data.forecasts` at most every `FORECAST_LOG_INTERVAL_MS` (30min — an in-memory clock short-circuits the common case so a 60s client poll never hits the store, and the blob is re-read immediately before writing to keep the read-modify-write window small). Once a horizon elapses, `scoreForecasts()` grades it against the reading nearest that horizon (±15min); a sensor gap marks it `unscorable` after an hour's grace so it isn't retried forever (same reasoning as a correction's `giveUp`). Sign convention: **`error = projected − actual`**, so *negative means the forecast sat below reality* (pessimistic / crying wolf) and positive means optimistic — the dangerous direction. Reports bias, MAE, % within 1.0/2.0, and — more important for a warning than the decimal place — **precision** (of the warnings raised, how many were followed by a real low) and **recall** (of the real lows, how many were warned), split by calibrated vs uncalibrated so a model change can be compared against its baseline. Needs `FORECAST_MIN_SCORED` (5) before it reports anything.

**What-if simulator** — `simulateActivity(type, minutes, intensity)` (backing `GET /api/simulate`, the "🔮 What if I…" card on the Activity tab with canned presets: walks, gym, heavy weights, golf, driving range, housework, play). Personal per-type profile when ≥2 logged sessions of that exact name exist (drop scaled sub-linearly `^0.8` by duration vs. `medianDuration`); otherwise `INTENSITY_CLASSES` defaults (light/moderate/vigorous), with the basis always named in the output and a wider uncertainty band (±1.2 vs ±0.8) for class-based estimates. Nets out windowed IOB/COB across activity + tail-to-nadir. Output: likely range, delayed-low risk, watch period, carb advice. Rising-profile types get the adrenaline-response answer instead.

**Preventative carb calculator** — `carbAdvice(projectedLow, settings, factor, {delayed})`: grams to lift a projected trough back to ~5.0 via the user's own ratios (`carbRatio`/`factor` g-per-mmol, generic "10–20g" when uncalibrated), classified by urgency — fast rescue (glucose tabs/juice) for imminent hypo, short buffer (banana/snack) otherwise, slower mixed carbs + carry-fast-carbs for delayed windows (`delayed: true` when nadir >2h out). Folded into the forecast card, simulator output, and workout advisor rather than being its own screen.

**Stacking caution** — `stackingCaution(data)` (returned on `GET /api/correction-factor`, red 🛑 box atop the correction card): fires when *any* dose sits in the 15–110min post-injection window with ≥0.5u total IOB — deliberately not just the newest dose, so a fresh meal bolus can't mask an earlier correction at peak. Doses <15min old are excluded (dosing again immediately is usually one combined decision, already handled by IOB subtraction). Unlike `coveredByIOB` (which replaces the suggested number), this shows even when the user types their own units — it targets the tired/frustrated rage-bolus moment. The retrospective version (stacking that actually produced lows) stays in `analysePatterns()`.

**Sensitivity map** — `sensitivityMap()` (backing `GET /api/sensitivity-map`, bar-chart card on Insights): clean resolved corrections bucketed by `SENS_BUCKETS` (morning/afternoon/evening/overnight) plus post-exercise (within 6h of a workout end) vs rest, every number shipped with its `n`. Needs ≥4 resolved corrections. `analysePatterns()` #16 surfaces the headline when two buckets with n≥3 differ ≥25%. Slices the app has no data for (sleep, alcohol, stress, meal fat) are deliberately absent — don't add them without adding the tracking first.

**Meal memory** — bolus entries carry an optional `mealName`, set only when logged via a meal-preset tap (manual carb edits clear the tag — `_selectedMeal` in app.js — so memory only aggregates genuine preset logs). `getMealMemory(name)` (`GET /api/meal-memory?name=`) computes per-meal outcomes over tagged logs with glucose coverage (≥2 needed): avg peak, median time-to-peak, return-to-range, low-risk-after, and a **delayed-rise flag** — the high-fat/protein signature, detected as a ≥2 mmol/L climb in the 2.5–6h window above the ~2.5h settle level (`mealOutcomes()`, shared with `analysePatterns()` #17's across-all-meals version). Rendered as the 📒 box when a preset is tapped. `suggestMealDose()` boosts same-`mealName` history 1.5×. "Best pre-bolus timing" is deliberately NOT computed — carbs and insulin share one timestamp, so the data doesn't exist.

**Simple ("tired") mode** — full-screen Now/Risk/Action view (`#simpleView`, 😴 floating toggle, persisted under `simpleMode` in localStorage): giant status-colored glucose + trend, forecast risk tier, and ONE action line synthesized in priority order — live alert > forecast-driven carb/recheck advice > overdue basal > "No action needed" (`simpleAction()`). Re-rendered by the same fetch cycle that feeds the full UI (`renderSimple()` hooks in fetchGlucose/loadAlerts/loadForecast), no separate polling.

**Pre-workout advisor** — `getWorkoutAdvice(type)` (backing `GET /api/workout-advice?type=`, rendered live in the Activity tab's Log-a-workout form as the type is picked/typed) answers "I'm about to do X": projected trough = trend-adjusted glucose − the type's median drop − what active insulin still has left to do (IOB × correction factor, only when the factor clears `MIN_PLAUSIBLE_FACTOR`). Risk tiers: trough <3.9 → `high` (suggests a concrete carb amount when both `carbRatio` and the factor are known: grams to lift the trough back to ~5.0, capped 5–30g), <5.0 → `caution` (carbs within reach), else `ok`. Types that historically *rise* get adrenaline-response advice (don't correct mid-session) instead of drop warnings. Unknown types (<2 logged sessions of that exact name) say so and push toward preset-consistent logging. Troughs below 3 print as "well below 3.9" rather than an arithmetically-negative glucose.

The post-exercise insight segments each workout into three phases - before (30min pre-*start*), during (start-to-end), and after (30-180min post-end) - and describes the actual trajectory (e.g. "dropped 4.0 during, then rose 2.7 after") rather than a single before/after delta, which misses what happens mid-activity (a workout can swing sharply one way during and recover the other way after, netting out looking like nothing happened). "Before" is anchored to the workout's *start*, not its end - anchoring to end-minus-30min was actually sampling the tail of the workout itself for anything longer than 30min. The "during" direction (drop vs rise) is whichever swung further from baseline; type is `warning` only for an in-activity drop or a delayed drop afterward (real hypo-risk signals), not for a rise during exercise (a normal adrenaline/glycogen response).

### Meal dose suggestion

`suggestMealDose(carbs)` (backing `GET /api/meal-suggestion?carbs=`) is a heuristic, not ML: it weights past insulin-dosed meals (`boluses` with `units>0 && carbs>0`) by carb-amount similarity, recency (~30-day half-life), time-of-day-bucket match, and exercise-proximity match, to get a base insulin:carb ratio. Before averaging, entries whose `ratio` (units/carbs) is more than 2.5x or less than 0.4x the matched group's median get their weight cut to 15% - a single atypical entry (e.g. a correction dose bundled into the same log as a meal, rather than logged separately) can otherwise dominate a small sample; this happened in practice (a 10g/6u entry, versus five other 10g entries all at 1-2.5u, pulled the suggestion for 10g up from ~1.9u to ~4.7u before this guard existed). It then nudges the (now outlier-resistant) ratio using how similar meals actually turned out — post-meal glucose nadir (using `correctionFactor` to translate an excessive drop into a unit reduction) or whether a correction was needed afterward (nudges up). If there isn't enough history yet (< 3 meals, or nothing similar enough), it falls back to `settings.carbRatio` (a manually-set "1 unit per Xg" ratio) when the user has set one.

On top of that, a correction add-on gets folded into whichever base suggestion was used (history-based or `carbRatio` fallback) — the same `(currentGlucose - idealTarget) / correctionFactor` calculation as the standalone correction suggestion, added via a shared `combine()` closure so all three return paths apply it uniformly. This mirrors how a real bolus calculator combines a "carb bolus" with a "correction bolus" into one number rather than treating them as two separate decisions.

### Meals without insulin

`boluses` entries don't require `units` — `POST /api/entries/bolus` accepts carbs-only entries (`units: 0`) for logging food eaten without dosing (e.g. treating/preventing a hypo). Frontend and history rendering treat `units === 0` as "carbs only" rather than a 0-unit dose. Anywhere historical meals feed an algorithm (`suggestMealDose`), carbs-only entries are excluded via the `units > 0` filter, but they still count as "recent carbs" for correction-interference/context checks.

### Backdated logging and editing

All three logging endpoints (`bolus`, `basal`, `correction`) accept an optional `time` (epoch ms or parseable date string) so a meal/dose can be logged after the fact instead of always defaulting to "now" — `resolveEntryTime()` clamps it to not-future. For a backdated correction, `startGlucose` is looked up via `glucoseAt()` (closest `glucoseHistory` point within 15min of that time, or the live cache if the time is within 6min of now) instead of always using the current cached reading.

Each entry type also has a `PATCH /api/entries/<kind>/:id` for in-place edits. Editing a correction's `units` or `time` invalidates its prior resolution (`resolved`/`actualGlucose`/etc. reset to null/false so `resolveCorrections()` re-derives them next glucose poll) — but time comparisons use a **60s tolerance**, not exact equality, because the frontend's `datetime-local` input only has minute precision; without the tolerance, saving an edit that didn't touch the time field would still spuriously "change" it by a few rounded seconds and wipe `startGlucose` every time. If you touch this code, keep that tolerance.

### Carbs on board (COB) and insulin on board (IOB)

Both are computed client-side in `public/app.js`, decaying independently per entry (`iobFraction()` / `cobFraction()`). `calcIOB()` sums over both `boluses` *and* `corrections` (a correction is still an insulin injection); `calcCOB()` only sums `boluses`, since corrections don't carry carbs. IOB uses the exponential insulin-action model (`iobFraction()`, same formula used by Loop/OpenAPS) with `IOB_PEAK = 75` / `IOB_DIA = 240` minutes, tuned for Novorapid — this replaced an earlier naive linear-decay model that wrongly treated a dose as 100% active immediately after injection.

**Carb absorption is shaped, and per-meal on the server.** COB was previously flat-linear over 180min for every meal, which paired badly with the exponential insulin curve: a constant carb trickle against insulin peaking at 75min produced an exaggerated mid-window dip in the forecast. A 60g meal covered by 6u at a 1u:10g ratio — i.e. exactly matched — projected a trough of 3.2 and fired a HIGH warning that was a pure modelling artefact (it now rises and settles at minimal). The absorption **rate** is now triangular (ramps to `peakMin`, tails to zero by `durationMin`), via `carbAbsorbedFraction()`; it integrates to exactly 1.0 and stays monotonic for any parameters, so no carbs are invented or lost. `buildCarbProfiles()` derives per-meal `peakMin`/`durationMin` for entries tagged with a `mealName`, from meal memory: **note the proxy** — what's measured is time-to-peak *glucose*, which lands later than peak absorption (glucose peaks where carb absorption and insulin action cross), so it's scaled down (×0.7) and clamped so one odd meal can't produce an absurd curve; delayed-rise meals get a long tail (≥300min), precisely the case flat-linear handled worst. `carbAbsorptionWithin()`/`carbsOnBoard()` take an optional profiles map, built once per forecast and reused across every simulation step.

Keep the client and server models in sync: if you tune `IOB_PEAK`/`IOB_DIA` or the carb curve, change **both** `lib/analysis.js` and `public/app.js`. The client COB tile uses the same triangular shape with generic `COB_PEAK = 60` / `COB_DURATION = 180`; the **per-meal** refinement is server-side only, since the browser has no access to meal memory. That divergence is deliberate and acceptable — the tile is informational, the forecast is what drives decisions.

The glucose chart's insulin-activity overlay (`iobActivityFraction()` in `drawChart()`) is a different curve from IOB's remaining-on-board fraction: it's the bell-shaped *rate of use*, peaking around `IOB_PEAK` minutes, derived as the per-minute finite difference of `iobFraction()` rather than a separately-derived formula, so it can't drift out of sync with the IOB model actually in use. `_iobActivityPeak` (a 1-unit dose's peak rate, sampled once at load) and `IOB_CHART_SCALE_UNITS = 10` calibrate the overlay's height so a typical dose fills a sensible fraction of the chart rather than being auto-scaled per-view (which would make the same dose look a different size depending on what else is visible). Doses are summed at each sample point, so overlapping boluses/corrections render as one merged curve rather than stacking separately.

### Settings (target range, insulin:carb ratio, body profile)

`GET`/`POST /api/settings` read/write `data.settings`. `targetLow`/`targetHigh` drive glucose color-coding and time-in-range everywhere (both backend insights/summary and the frontend chart band/dot colors) — there's no hardcoded 4–10 range left. `idealTarget` is a separate single-value target (distinct from the low/high range) that correction and meal-dose suggestions aim for. `carbRatio` is the manual meal-suggestion fallback described above. The frontend fetches settings once in `init()` before the first render so colors are correct on load, and again whenever the Settings tab is opened.

`heightCm`/`weightKg`/`sex`/`bodyFatPct` (Settings tab "Body Profile" card) are stored purely as context for the Insulin Health Check below (BMI, dose-per-kg) — **never read by any suggestion or calculation elsewhere** (`suggestMealDose`, correction factor, etc.). This is a deliberate boundary: self-estimated body fat % in particular isn't precise enough to weight into anything without giving false confidence. If you're tempted to use these fields for more than display, reconsider.

### Meal presets

`data.mealPresets` (`{id, name, carbs}`, `GET`/`POST /api/meal-presets`, `DELETE /api/meal-presets/:id`) are the user's own regular meals (defaults to just `Coffee` = 10g), shown as quick-select buttons on the Log a meal card instead of generic round-number carb amounts (which the actual user of this app doesn't use) — tapping one fills the carbs field the same way the old hardcoded buttons did. Managed from the Settings tab; `loadMealPresets()` (Track tab buttons) and `loadPresetManager()` (Settings tab list) are separate render functions but both re-fetch from the same endpoint, so adding/removing a preset in Settings is reflected on Track next time either runs.

`data.workoutPresets` (`{id, name}`, same `GET`/`POST`/`DELETE` shape under `/api/workout-presets`) is the same idea for the Activity tab's "Log a workout" form, but name-only - there's no fixed "amount" for a workout the way there is carbs for a meal. This exists specifically to keep workout-type naming consistent (e.g. "Strength workout" vs "Strength training" typed on different days), since `analysePatterns()`'s exercise-day detection and post-workout insight group activities by exact `workoutType` string match - inconsistent free-text naming silently fragments what should be one pattern into several. Both quick-select rows share the `.qc` CSS class for pill-button styling, but their "clear the .on state" handlers are scoped to their own container (`#qc` / `#workoutQc` respectively) rather than querying `.qc button` globally, so tapping a meal preset doesn't clear a workout preset's selected state or vice versa.

### Insulin Health Check

`checkInsulinHealth()` (backing `GET /api/insulin-health`, shown on the Insights tab) reports standard clinical-style dose metrics over the trailing 7 days: Total Daily Dose (TDD = bolus + correction + basal), TDD/kg (needs `weightKg`), bolus:basal split, time-in-range, TBR/TAR (% below 3.9 / above 10.0 — the **fixed** ADA/ATTD consensus cut-points, deliberately not `targetLow`/`targetHigh`, so they stay comparable if the personal range changes; targets <4% and <25%, tiles color red/orange beyond them), CV (≤36% target), and BMI (needs `heightCm`+`weightKg`). `getDailySummary()` carries the same TBR/TAR/CV trio for today only. `periodStats()` computes these for an arbitrary window; the function calls it twice (this week vs. the prior week) and emits a plain-language note for each metric that moved meaningfully (TDD ±2u/day, TIR ±5pp, bolus share ±8pp) — silence on a metric means it didn't move enough to be worth mentioning, not that it's broken. Requires ≥20 glucose readings in the trailing week before returning anything (`available: false` otherwise), matching `analysePatterns()`'s data-sufficiency bar.

### CSV export

`GET /api/export.csv` flattens all five record arrays into one wide sheet with a `type` discriminator column (`bolus`/`carbs_only`/`basal`/`correction`/`workout`/`daily_summary`/`glucose`), sorted by time. Triggered from the frontend via a plain `<a href download>` — auth works because the browser sends the existing session cookie on that same-origin navigation, no token wiring needed.

### Frontend

`public/app.js` polls the backend directly (no client-side router/state library). The glucose trend chart is drawn to a `<canvas>` by hand in `drawChart()`, including bolus/correction/exercise event markers pulled from `GET /api/glucose-history`. Hour labels on the x-axis are anchored to actual clock-hour boundaries (`x(hourBoundaryTime)`), not to wherever a data point happens to fall — the latter breaks down when readings are gapped (sensor/network lag), since two adjacent hours can each have their only representative point land right next to the boundary and render almost on top of each other.

The exercise event's shaded width converts `duration` (minutes) to pixels via `duration*60000/tRange*cW` - i.e. relative to the chart's actual visible time span, not the nominal range-button value (`chartHours`), which don't necessarily match and previously produced a width so small it always hit the 20px floor regardless of how long the workout actually was. The band is also clamped to the plot's right edge so a just-finished workout doesn't paint into the axis padding.

UI conventions worth keeping: inputs are 16px (anything smaller triggers iOS Safari's auto-zoom on focus); destructive taps (`delEntry`, `delWorkout`, preset removes) go through `confirm()`; transient feedback (settings saved, API errors) uses the fixed-position `toast()` rather than inline per-card messages (the old inline `#settingsMsg` was invisible when saving from the Body Profile card); keystroke-driven suggestion lookups (`updateMealSuggestion`/`updateCorrSuggestion`) are debounced 300ms so typing "45" doesn't fire a request for "4"; the 60s refresh interval includes `fetchEntries()` and `loadAlerts()` so a correction that resolves server-side (or a post-workout drop window opening) shows up without a manual action.

List presentation: the entry log groups under day headers (Today/Yesterday/date) with clock-only times per row; the log, Recent Workouts, and Correction History all cap at a screenful with a dashed "Show more" expander (`expandedLists` state + `showMoreBtn()`); Pattern Analysis renders in three severity sections (Needs attention / Going well / Worth knowing) so warnings always surface first. The glucose card deliberately shows only the value, trend arrow, and delta chip — the old "Falling slowly · 1h 4m ago" meta line was removed as noise (the age mostly reflected LibreLinkUp's documented follower lag). Live alerts render in `#alertsBox` directly under the glucose card, where logging decisions actually happen.

**Installability & navigation**: the app ships `manifest.json` + generated PNG icons (`public/icon-*.png`, `apple-touch-icon.png` — regenerate with the scratch script's raw-PNG encoder if the design changes) and the iOS standalone metas, so it installs to the home screen full-screen. **Deliberately no service worker** — an offline cache of a glucose app is a stale-data trap; if there's no connection, showing nothing beats showing yesterday's numbers as if live. Because standalone mode has no browser reload button, the glucose card carries the ↻ `refreshAll()` control (spin + "Updated" toast), and a `.g-stale` warning escalates with age: past 30min it's just "reading is ~Xm old"; past 40min with nothing new it's more likely a sensor gap (warmup/change/out-of-range) than follower lag, so it says so and points at the primary Libre app. Age comes from `readingMs`, not a re-parse of the raw timestamp string — see `parseLibreTimestamp()`. Navigation is a **fixed bottom tab bar** (thumb reach; `env(safe-area-inset-bottom)` padded, `viewport-fit=cover` in the meta) — the FAB and toast sit above it. Tab switches scroll to top. The Track tab opens with the sticky `.quick-log` bar (Meal/Correction/Basal → `jumpTo()` scrolls + focuses); dose/carb inputs have ± steppers (`stepInput()`, 0.5u/5g/1u steps) whose dispatched input events are flagged `_stepping` so a portion nudge doesn't untag a preset meal; both suggestion boxes have one-tap "Use" buttons.

**Forecast on the chart**: when the last reading is <45min old, `drawChart()` extends the x-domain `horizonHours` past it and draws the dotted risk-colored **simulated curve** point-by-point from `_lastForecast.path` (falling back to a straight line if no path) — the shape, a dip then recovery, is the whole reason for simulating rather than projecting a single endpoint. The path is anchored at `Date.now()` (where the simulation starts), not the last reading, so the drawn curve lines up with the minutes actually modelled. The hollow marker + label sit on the **trough** (`troughAtMin`), the number being warned about, not the far end of the horizon. The y-axis floors at 2 and the drawn endpoint is clamped to it, so a projection near the floor can't paint off-canvas into the time labels; the label reads `<3` rather than an implausibly low number. The extension is independent of scrubbing so crosshair geometry never shifts mid-drag; `loadForecast()` repaints the chart from cache so the curve tracks each new forecast.

**Urgent hypo nudge**: the forecast card and glucose number are passive — you have to be looking. `checkUrgentAlert()` (called from both `fetchGlucose()` and `loadForecast()`, since each supplies half the picture) fires a two-tone WebAudio beep + vibration when glucose is **actually** below `targetLow` (which outranks a forecast — it's happening, not projected) or the forecast risk is `high`. Re-fires at most every 10min while a condition holds, and immediately if the condition changes. Sound is opt-out via **Settings › Alert Sound** (`alertSound` in localStorage, `renderAlertSeg()`); vibration still fires when muted. Tapping "On" plays the tone once — immediate confirmation, and it doubles as the user gesture that unlocks the AudioContext on mobile.

**Theming**: light/dark via CSS vars on `:root` / `html.dark`; `applyTheme()` follows `prefers-color-scheme` in `auto` mode (an explicit Light/Dark choice persists in localStorage under `theme`, segmented control in Settings). The canvas chart can't inherit CSS vars — `drawChart()` reads every color through `cssVar()` at draw time and `applyTheme()` triggers a redraw from `window._chart` (the cached last-draw state). The glucose status trio (red/green/orange) is CVD/contrast-validated on both surfaces; dark mode only swaps red to `#ef4444` — don't "brighten" the others, brighter steps fail the dark-surface lightness band.

**Chart interactivity**: `window._chart` caches data + scale closures from the last draw; the scrub layer (pointer events, `touch-action: pan-y` so vertical scrolling survives) redraws with `{scrubIdx}` for the crosshair and positions the HTML `#chartTip` (HTML, not canvas text, so it's crisp at any DPR). The `#liveDot` pulse marker sits on the last reading and hides while scrubbing. **A hidden panel measures 0×0** — `drawChart()` bails on zero rects instead of wiping the canvas/cache, and `showTab('track')` repaints from cache; if you add another draw trigger, keep that guard in mind. The wipe-reveal animation only runs on explicit range switches/first load (`loadChart(true)`), never the 60s refresh.

**Motion & viz conventions**: glucose value tweens between readings (`tweenNumber`) with a pop; tabs use a sliding glider pill (`moveGlider()`, repositioned on resize/theme/tab change); panels stagger their cards on open; first-load placeholders use `.skel` shimmer (`dataset.loaded` guards re-flashes on background refreshes); horizontal swipe switches tabs everywhere except inside `.chart-wrap` (the chart owns horizontal drags); everything respects `prefers-reduced-motion`. TIR meter bars (Today: personal range; Insulin Health Check: fixed clinical thresholds, normalized) render segments at width 0 and animate via `animateTirBars()`, with 2px gaps and a labeled legend so identity is never color-alone. IOB/COB tiles carry decay drain-bars + "peaks in Xm / gone by HH:MM" microtext derived from the same client-side models.

`updateBasalStatus()` (called from `render()`, so it ticks every 30s independent of any refetch) shows a countdown to the next Lantus dose (24h since the last one) and switches the whole basal card into a pulsing red `.basal-overdue` state once that window has passed — deliberately more attention-grabbing than the rest of the UI, since a missed basal dose matters more than most other logging gaps.

### External integration: Apple Health

`POST /api/health` accepts two different payload shapes and auto-detects which one it got:
- **Health Auto Export** (recommended, see `README.md`) sends `{ data: { metrics: [{name, units, data:[{date, qty|Avg/Min/Max, ...}]}], workouts: [...] } }`. Energy values are in **kJ** and get converted to kcal (`kJtoKcal()`). Daily metrics (`active_energy`, `apple_exercise_time`, `apple_stand_hour`, `step_count`, `resting_heart_rate`) are iterated per-day and upserted via `upsertDailySummary()`, which merges into whichever `daily_summary` activity matches that calendar day rather than assuming everything is "today" — a single export/automation run can carry multiple days. Workouts carry `avgHeartRate`/`maxHeartRate` (stored, surfaced in the Activity tab and in the post-exercise insight text), but per-minute time series (`heartRateData`, `heartRateRecovery`, etc.) are deliberately *not* stored — a single export can be hundreds of KB and only the summary stats are useful here.
- **Legacy Apple Shortcuts** format `{ summary: {...}, workouts: [{startTime, endTime, ...}] }` is still supported for anyone not using Health Auto Export.

Workout dedup in both paths is by matching the stored `startTime` field (`w.start` for Health Auto Export, `w.startTime` for Shortcuts) against existing `activities`.

**Apple's free Shortcuts actions cannot query Workout objects at all** — "Find Health Samples" only exposes quantity/category samples (heart rate, steps, etc.), not workouts. This is a real platform limitation (confirmed via research, not a bug in this app), not something the original Shortcuts-based README instructions could have ever actually accomplished. Getting workouts into Shortcuts requires a paid third-party action (e.g. Toolbox Pro's "Get Workouts"). Because of this, `POST /api/activities/workout` (+ matching `PATCH`/`DELETE .../:id`) exists as a manual-entry fallback, surfaced as a "Log a workout" form on the Activity tab. Manually-logged workouts are flagged `manual: true` and are the only ones with a delete button in the UI (synced entries aren't locally deletable since they'd just reappear on the next sync).

Without an explicit backdated time, `POST /api/activities/workout` assumes the workout *just ended* (`start = now - duration`), not that it's starting now — someone logging a completed workout is almost always doing so shortly after finishing it, and stamping it as starting now would plot it as happening in the future relative to when it actually occurred, throwing off its position on the glucose chart relative to the glucose reading at the time. Editing `duration` via `PATCH` recomputes `endTime` from the *existing* `time` (start), so it doesn't go stale relative to the new duration.

### Known LibreLinkUp data lag (not a bug in this app)

Follower-account data (what this app polls) can genuinely lag the primary sharer's own LibreLink app by up to 15–20 minutes, and real new readings can arrive well over 5 minutes apart even though `fetchGlucose()` polls every `POLL_MS` — this is a documented characteristic of LibreLinkUp's sync, not something fixable by polling more often (and `POLL_MS` shouldn't be lowered regardless, see above). If glucose values look "wrong" compared to the primary app, check whether the underlying reading timestamps genuinely differ before assuming a bug here.
